# Orchestrator 실행 런북

> `/prompts/orchestrator.md`를 기준으로 100화 웹소설을 생성할 때 사람이 그대로 따라 하는 체크리스트다. 각 단계는 **준비 → 기획(1~8번) → 작성 루프(9~11번) → 분기점 점검 → 상태 갱신** 순서로 진행한다.

## 0. 실행 전 준비

- [ ] `/prompts/orchestrator.md`를 실행 기준 문서로 정한다.
- [ ] `/prompts/lock.md`, `/prompts/prompt_1_user_input_organizer.md` ~ `/prompts/prompt_11_handoff.md`, `/prompts/promptpack_toc.md`가 있는지 확인한다.
- [ ] 사용할 모델/도구(Codex, Claude Code, Cursor 등)에 repo 파일을 읽을 수 있는 권한이 있는지 확인한다.
- [ ] 진행 방식을 정한다.
  - **AUTO**: 필수 확인 지점에서만 사용자에게 묻고 계속 진행한다.
  - **STEP**: 각 모듈 결과를 사용자에게 보여주고 승인받는다.

## 1. 사용자 입력 자료 준비

### 1-1. `source/raw_memo.md` 작성

아래 내용을 자유 형식으로 적는다. 모르면 `미정`으로 둔다.

- [ ] 작품 한 줄 아이디어
- [ ] 장르, 목표 독자, 플랫폼 감각
- [ ] 주인공의 욕망, 결핍, 절대 하지 않는 행동
- [ ] 1~5화 안에 반드시 보여줄 사건
- [ ] 원하는 재미: 통쾌함, 긴장감, 감정선, 관계성
- [ ] 피하고 싶은 소재, 전개, 문체

### 1-2. `reference/like` 자료 준비

- [ ] 좋아하는 작품/회차 원문 또는 직접 작성한 예시를 `reference/like/`에 넣는다.
- [ ] 문체, 대사 리듬, 장면 전환, 감정 표현, 엔딩 훅이 보이는 샘플을 우선 넣는다.
- [ ] 각 파일 상단에 “왜 좋은지”를 1~3줄로 적는다.

### 1-3. `reference/dislike` 자료 준비

- [ ] 피하고 싶은 문체, 대사, 전개 샘플을 `reference/dislike/`에 넣는다.
- [ ] 반복 설명, 감정 과잉/부족, 전개 지연, 비약 등 싫은 이유를 표시한다.
- [ ] “절대 금지할 규칙”을 1~3줄로 적는다.

## 2. orchestrator 시작

- [ ] 모델/도구에 `/prompts/orchestrator.md`를 먼저 부착한다.
- [ ] orchestrator 지시에 따라 LOCK과 1~11번 프롬프트를 순서대로 부착한다.
- [ ] `source/raw_memo.md`, `reference/like/`, `reference/dislike/`를 읽게 한다.
- [ ] 시작 메시지에 진행 모드(AUTO 또는 STEP)를 명시한다.
- [ ] orchestrator가 버전 누락, 표준 외 파일, 충돌 자료를 보고하면 본문 생성을 시작하지 말고 정리한다.

## 3. 1~8번 기획 단계

- [ ] **1번 사용자 입력 정리**: raw memo와 reference 자료를 정리하고 핵심 4개 게이트를 확정한다.
  - 지향 문체
  - 대사/서술 비율
  - 감정 표현 수위
  - 금지 문체
- [ ] **2번 트렌드 조사**: 장르·플랫폼 트렌드를 조사하고 적용 가능한 신호만 남긴다.
- [ ] **3번 트렌드 선택**: 사용할 트렌드와 버릴 트렌드를 확정한다.
- [ ] **4번 상업 콘셉트**: 독자 보상, 차별화, 초반 판매 포인트를 정한다.
- [ ] **5번 세계 설계**: 세계관 규칙, 권력 구조, 사건이 발생하는 조건을 정한다.
- [ ] **6번 캐릭터 설계**: 주인공·주요 인물·캐릭터 불변 원칙을 정한다.
- [ ] **7번 전체 아크**: 100화 구조와 25/50/75/100화 분기점을 잡는다.
- [ ] **8번 초반 설계**: 1~5화 설계와 각 회차 비트 시트를 만든다.
- [ ] 8번 결과를 확인한 뒤 1화 본문 작성으로 넘어간다.

## 4. 9~11번 회차 작성 루프

1화부터 100화까지 아래 루프를 반복한다.

### 4-1. 9번 본문 작성

- [ ] 현재 회차 번호를 확인한다.
- [ ] 1~5화는 8번 비트 시트를 사용한다.
- [ ] 6화 이후는 직전 11번 인계의 다음 회차 비트 시트를 사용한다.
- [ ] LOCK, 지향/회피 규칙, 캐릭터 불변 원칙을 상단 기준으로 유지한다.
- [ ] 본문 출력 후 자체 승인 단계 없이 즉시 10번 진단으로 넘긴다.

### 4-2. 10번 진단

- [ ] 해당 회차 본문을 1화 단위로 진단한다.
- [ ] 사건 전진 실패, 캐릭터 불변 위배, 시그니처 위반을 핵심 적색으로 본다.
- [ ] 핵심 적색이 있으면 같은 회차를 재작성한다.
- [ ] 비핵심 적색은 다음 회차 지침으로 넘긴다.
- [ ] 비트 누락이 있으면 사용한 비트 ID와 원래 비트 시트를 대조한다.

### 4-3. 11번 인계

- [ ] 통과한 회차의 핵심 사건을 요약한다.
- [ ] 캐릭터·관계 변화와 미해결 후폭풍을 갱신한다.
- [ ] 새 복선과 회수된 복선을 구분한다.
- [ ] 다음 회차 비트 시트를 만든다.
- [ ] 5화 단위마다 직전 5화 변화 압축을 갱신한다.

## 5. 25/50/75/100화 점검

분기점 회차를 통과하면 작성 루프를 잠시 멈추고 점검한다.

- [ ] **25화**: 1부 목표 달성/실패, 주인공 변화, 초반 복선 회수 상태를 확인한다.
- [ ] **50화**: 중반 동력, 적대 세력 확장, 관계 압력, 반복 패턴을 확인한다.
- [ ] **75화**: 후반 진입 명분, 장기 복선 회수 계획, 결말 압력을 확인한다.
- [ ] **100화**: 최종 회수, 캐릭터 결산, 남은 떡밥, 후속 확장 가능성을 확인한다.
- [ ] 아크 수정이 필요하면 7번으로 돌아가 청사진을 조정한다.
- [ ] 조정한 청사진은 8번 또는 11번에서 다시 비트 시트로 환원한 뒤 작성 루프로 복귀한다.

## 6. `output/state` 갱신 방식

상태 파일은 매 회차 “다음 판단에 필요한 사실”만 짧게 갱신한다. 본문 전체를 복사하지 않는다.

| 파일 | 언제 갱신하나 | 적는 내용 |
|---|---|---|
| `current_state.md` | 매 모듈 종료 | 현재 단계, 회차, 마지막 완료 작업, 다음 작업 |
| `decisions_log.md` | 결정 발생 시 | 사용자 확정 사항, 되돌리면 안 되는 기획 결정 |
| `open_questions.md` | 답이 필요한 질문 발생 시 | 다음 사용자 호출 때 물어볼 질문과 이유 |
| `continuity_ledger.md` | 설정/사건 확정 시 | 이후 회차에서 바뀌면 안 되는 사실 |
| `foreshadowing_ledger.md` | 복선 생성/회수 시 | 복선 ID, 심은 회차, 회수 예정/완료 상태 |
| `character_pressure_ledger.md` | 인물 선택 발생 시 | 인물별 욕망, 압박, 선택, 관계 변화 |
| `retry_counter.md` | 적색/재작성 발생 시 | 재시도 횟수, 원인, 조치, 결과 |
| `signature_log.md` | 지향 시그니처 반영 시 | 반영 위치와 근거 |
| `anti_pattern_log.md` | 회피 패턴 발견 시 | 발생 위치, 문제, 다음 회차 방지 지침 |
| `comparison_reference_index.md` | 참고 자료 추가/정리 시 | like/dislike 자료 위치와 비교 기준 |

## 7. 종료 체크

- [ ] 100화 본문 작성이 끝났다.
- [ ] 100화 점검에서 핵심 미회수 복선이 없다.
- [ ] 캐릭터 불변 원칙 위배가 없다.
- [ ] `output/state`의 열린 질문과 재시도 항목이 정리됐다.
- [ ] 최종 산출물 위치와 남은 후속 작업을 사용자에게 보고한다.
